# Create Issue

ユーザーの依頼内容とコードベースの調査をもとに、プロジェクト規約に準拠したGitHub Issueを作成する。

## ワークフロー

### 1. リポジトリの特定

引数や会話でリポジトリが明示されていればそれを使用。未指定の場合:

1. 現在の作業ディレクトリのリモートを確認
   ```bash
   git remote -v
   ```
2. 対象のリポジトリが意図通りかを AskUserQuestion で確認する

### 2. コードベースの調査

Issueの内容を充実させるため、関連するコードを確認する。

調査する内容:
- 関連ファイル・コンポーネントの構造
- 既存の実装パターン
- DBスキーマや型定義（実現方法の記述に活用）

### 3. Issueの内容を構成

以下のテンプレートに従いIssueを構成する。

```markdown
## 概要

背景と目的を踏まえたIssueの概要を簡潔に記述する。

## 背景

このIssueの背景となっている問題について記述し、具体的な課題に落とし込めている場合はその課題も記述する。

## 目的

なぜこのIssueに取り組むのか、背景を踏まえて記述する。

## 実現方法

実現方法を簡易的にまとめて記述する。

### ToDo

- [ ] <対応手順1>
- [ ] <対応手順2>
```

記述の方針:
- コードベース調査で得た情報（テーブル名・型名・ファイルパスなど）を具体的に盛り込む
- ToDoは実装ステップが明確になるよう細かく分解する
- 見出しはh2（##）から始める
- 箇条書きで文章の流れを表現しない
- 文章はコロン（:）で終わらせず句点で終わらせる

### 4. Issueの作成

```bash
gh issue create \
  --repo <owner>/<repo> \
  --assignee NAKKA-K \
  --title "<タイトル>" \
  --body "$(cat <<'EOF'
<Issue本文>
EOF
)"
```

タイトルの方針:
- ユーザーの依頼内容を簡潔にまとめる
- 体言止めまたは「〜する」形式
- 80文字以内

### 5. 完了通知

```bash
afplay /System/Library/Sounds/Glass.aiff
```

作成したIssueのURLをユーザーに報告する。

## 重要なルール

- Assignees は必ず `NAKKA-K` を設定する（`--assignee NAKKA-K`）
- URLはコードスパンで囲まず平文で記載（リンクとして機能させる）
- 見出しはh2（##）から始める
- 数字の箇条書き（1. 2. 3.）や**太字**で見出しを表現しない

## エラーハンドリング

| 状況 | 対処 |
|------|------|
| リポジトリ不明 | AskUserQuestionで確認 |
| gh CLI認証エラー | `gh auth status` で確認、`gh auth login` を案内 |
| 関連コードが見つからない | 調査結果を伝えたうえでユーザーが提供した情報のみでIssueを作成 |
