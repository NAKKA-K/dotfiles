# Review Fixes

GitHubのPull Requestの未解決レビューコメントを取得し、ユーザーが選択した修正をコードに適用してコミットする。

## ワークフロー

### 1. PR番号の特定

引数でPR番号が指定されていればそれを使用。未指定の場合:

```bash
gh pr view --json number -q .number
```

取得できない場合はAskUserQuestionでユーザーに確認。

### 2. レビューコメントの取得

```bash
gh pr view <PR番号> --json reviews,reviewThreads
```

未解決（unresolved）のスレッドのみを対象に以下を抽出:
- ファイルパスと行番号
- コメント内容（prefixも含む）
- コメントのprefix判定: `[must]` は必須修正、`[imo]` `[nits]` はオプション

### 3. 修正項目の選択

未解決コメントをユーザーに提示した後、AskUserQuestionで修正する項目を選択してもらう:

```
multiSelect: true
各オプション: "[ファイル名:行番号] コメント内容の要約"
```

`[must]` タグのものは優先度が高いことを明示する。

### 4. 修正の実施

選択された各項目について:

1. `Read` ツールで対象ファイル全体を読み込む（既存パターンを把握する）
2. レビューコメントの指摘を正確に理解する
3. CLAUDE.md の規約に従った修正方針を決定する
4. `Edit` ツールで修正を適用する（最小限の変更のみ）
5. 関連する修正をまとめてコミット（性質が異なる修正は分離）

コミットメッセージ例:
```
refactor: レビュー指摘事項の修正
```

### 5. 品質チェック

CLAUDE.md の品質チェックセクション、または `Makefile`・`package.json` のスクリプトを確認して実行する。不明な場合はユーザーに確認する。

エラーが発生した場合は修正内容を再検討してユーザーに相談する。

### 6. 完了通知

```bash
afplay /System/Library/Sounds/Glass.aiff
```

ユーザーへの報告内容:
- 修正した項目数
- 変更ファイル一覧
- コミットハッシュ
- 次のステップ（`git push` が必要な場合はその旨を案内）

## 重要なルール

- **ファイルは必ず全体を読んでから修正する**（既存パターンの把握が必須）
- **最小限の変更のみ**（レビュー指摘箇所以外は変更しない）
- **`git push` はユーザーの承認を得てから行う**（permission が ask 設定のため）

## エラーハンドリング

| 状況 | 対処 |
|------|------|
| PR番号取得失敗 | AskUserQuestionで番号を確認 |
| レビューコメント取得失敗 | `gh auth status` で認証確認 |
| ファイルが見つからない | ブランチが最新か確認し必要なら pull |
| テスト失敗 | 修正内容を再検討してユーザーに相談 |
